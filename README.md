# ZoomClone — Video Conferencing Platform

A functional clone of Zoom's web meeting platform: create instant meetings,
join by ID/link, schedule meetings, and hold real peer-to-peer video calls —
built for an SDE Fullstack assignment.

## Tech Stack

- **Frontend:** Next.js (App Router, JavaScript), Tailwind CSS
- **Backend:** Python, FastAPI
- **Database:** SQLite via SQLAlchemy ORM
- **Real-time video:** Native browser WebRTC (RTCPeerConnection), with a
  FastAPI WebSocket used purely as the signaling relay

## Assumptions Made

- No login/authentication — a single default user ("Default User") is
  assumed logged in at all times, per the assignment's "No Login Required" note.
- Meetings are mesh WebRTC (every participant connects directly to every
  other participant) — appropriate for small meetings (2-6 people). See
  "Architecture Decisions" below for why this doesn't scale further and
  what would need to change.
- The database is seeded with a few sample scheduled/recent meetings on
  first run so the dashboard isn't empty when first demoed.
 Note: npm audit flags some Next.js 14.x advisories not applicable to this app's actual feature usage (e.g. next/image caching, i18n middleware) — upgrading to Next.js 16 was deprioritized given the assignment timeline, since it involves breaking routing/config changes.
## Database Schema

Two tables with a one-to-many relationship:

```
meetings                          participants
─────────────────────             ─────────────────────
id (PK)                           id (PK)
meeting_code (unique)             meeting_id (FK -> meetings.id)
title                             display_name
description                       joined_at
host_name                         left_at (nullable)
meeting_type (instant/scheduled)
scheduled_time (nullable)
duration_minutes (nullable)
status (scheduled/active/ended)
created_at
```

**Why `meeting_code` is separate from `id`:** `id` is an internal
auto-increment primary key; `meeting_code` is the public-facing identifier
used in shareable links (`/meeting/{code}`). Decoupling these means the
public code could be rotated/regenerated without touching any foreign keys
elsewhere in the schema.

**Why `participants` is its own table, not a JSON column on `meetings`:**
It gives us real attendance history per meeting (needed for a genuine
"recent meetings" feature) and sets up cleanly for host-controls features
like "remove participant" — you can't target a specific row to delete if
attendees are just an unstructured JSON blob.

## Architecture Decisions (read this before the evaluation interview)

**Why WebRTC signaling via WebSocket, not polling:**
WebRTC connection setup (exchanging SDP offers/answers and ICE candidates)
is a real-time handshake between two browsers — polling would add latency
and complexity for no benefit. A WebSocket lets the server push these small
JSON messages the instant they're available, and FastAPI supports
WebSockets natively without extra libraries.

**Why the signaling server never touches video/audio:**
Once two browsers exchange SDP/ICE info through the WebSocket relay, they
establish a *direct* peer-to-peer connection for actual media — the server's
job ends at introductions. This is why the backend stays cheap to run even
during an active call: it's relaying a handful of small text messages, not
streaming video.

**Why this doesn't scale past ~4-6 participants (and what would fix it):**
This app uses **mesh topology** — every participant holds a direct
connection to every other participant. Each participant's upload bandwidth
therefore multiplies with room size (with N participants, everyone uploads
their video N-1 times). Real Zoom/Google Meet solve this with an **SFU**
(Selective Forwarding Unit) — a media server that each participant uploads
to *once*, which then forwards streams to everyone else. Implementing an
SFU (e.g. with mediasoup or LiveKit) would be the natural next step to
support larger meetings.

**Why offer/answer direction is one-way (only new joiners initiate):**
If both sides tried to create offers to each other simultaneously, WebRTC
handles that badly ("glare"). Instead: when you join, you learn who's
already in the room and initiate connections to each of them. Existing
participants stay passive and simply answer when your offer arrives — a
deterministic handshake direction with no race condition.

## Local Setup

### Backend
```bash
cd backend
python -m venv .venv
# Windows: .venv\Scripts\activate | Mac/Linux: source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```
Backend runs at `http://localhost:8000`. Visit `http://localhost:8000/docs`
for interactive API docs (auto-generated by FastAPI).

### Frontend
```bash
cd frontend
npm install
cp .env.local.example .env.local
npm run dev
```
Frontend runs at `http://localhost:3000`.

### Try it
1. Open two browser tabs/windows (or one normal + one incognito, since
   camera/mic permission and WebRTC work per-tab).
2. In tab 1, click **New Meeting** — you'll be dropped into a meeting room
   with a generated meeting ID.
3. Copy the meeting link/ID from the URL.
4. In tab 2, click **Join Meeting**, paste the ID/link, enter a display
   name, and join — you should see and hear both tabs live.

## Deployment

### Backend → Render
1. Push this repo to GitHub.
2. On [render.com](https://render.com), click **New → Web Service**, connect
   your GitHub repo, set root directory to `backend`.
3. Build command: `pip install -r requirements.txt`
4. Start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
5. Render's free tier supports WebSocket connections, so the signaling
   endpoint (`/ws/{meeting_code}`) will work as-is.
6. Once deployed, copy the Render URL (e.g. `https://zoomclone-api.onrender.com`).

### Frontend → Vercel
1. On [vercel.com](https://vercel.com), import the same GitHub repo, set
   root directory to `frontend`.
2. In Vercel's project settings → Environment Variables, add:
   `NEXT_PUBLIC_API_URL` = your Render backend URL (from above)
3. Deploy. Vercel auto-detects Next.js and handles the build.

### Note on CORS
`backend/app/main.py` currently allows all origins (`allow_origins=["*"]`)
for ease of local development. For a tighter production setup, replace
this with your actual deployed Vercel domain.
